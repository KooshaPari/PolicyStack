## Policy Wrapper Executables

This folder contains reference implementations for cross-language policy evaluators that
consume the canonical policy-wrapper payload:

- `agent-scope/policy_wrapper.schema.json`
- `policy-wrapper-rules.json` generated by sync output (`policy_wrapper` block)

### Command protocol (common)

All wrappers accept the same arguments:

- `--bundle <path>` path to policy wrapper JSON
- `--command <text>` command string to evaluate
- `--cwd <path>` working directory for git-based condition checks (defaults to `cwd`)
- `--json` emit JSON verdict

The wrapper returns one of:

- `allow`
- `request`
- `deny`

For convenience, wrappers skip unmatched rules and default to `allow`.

The manifest written by `sync_host_rules.py` also exposes wrapper fallback policy:

- `fallback_missing_policy`: value used when no binary is present or usable
- `fallback_malformed_bundle`: value used when bundle load/parse fails
- `fallback_condition_eval_error`: value used when condition evaluation fails
- Supported values are `allow`, `request`, or `deny`.

The JSON bundle is always emitted in the same `policy_wrapper` schema shape:

- `schema_version`: integer schema tag
- `required_conditions`: sorted required condition names needed by conditional rules
- `commands`: list of conditional command rules (must include matcher, pattern, and action)

### Recommended integration

The canonical runtime path is:

1. Write host artifacts with conditional rules included:

```bash
python policy-contract/scripts/sync_host_rules.py \
  --policy-json /tmp/effective-policy.json \
  --out-dir /tmp/policy-host \
  --include-conditional
```

2. Read `policy-wrapper-dispatch.manifest.json` from the same folder.
3. Use `policy-wrapper-dispatch.sh` as the last-mile conditional gate.

Host hook pseudocode:

```
# existing allow/deny check passes on a command
if platform_decision == "allow" {
  dispatch_result=$(policy-wrapper-dispatch.sh --json --bundle /tmp/policy-host/policy-wrapper-rules.json --command "$command")
  case "$(echo "$dispatch_result" | jq -r .decision)" in
    allow)   continue ;;
    request) prompt_user ;;
    deny)    block ;;
  esac
}
```

The dispatch script defaults to `missing_policy_default` (default `allow`) when a
binary is unavailable.
Use `--missing-policy-default request` if you prefer safer fail-closed behavior:

```bash
policy-wrapper-dispatch.sh \
  --bundle /tmp/policy-host/policy-wrapper-rules.json \
  --command "git checkout main" \
  --missing-policy-default request
```

If you do not have `jq`, parse JSON with your host language of choice or run without
`--json` for a single decision token.

### Platform support and migration model

This repository intentionally keeps wrapper enforcement as the conditional layer:

- Static command gates (`allow`/`deny`/`request`) are directly exported for Cursor,
  Claude, Codex, and Droid where those host surfaces exist.
- Conditional evaluations (`git_is_worktree`, `git_clean_worktree`, etc.) require
  `policy-wrapper-dispatch` at execution time.

Manifest + runtime fallback model:

- If no wrapper binary is available, policy dispatch falls back to
  `missing_policy_default` (default `allow`).
- The manifest includes `fallback_malformed_bundle` and
  `fallback_condition_eval_error` for explicit manifest consumers that want policy
  values for malformed bundle or eval-failure cases.
- `policy-wrapper-dispatch.sh` routes failures by reason:
  - missing/unusable binary and binary health failures use `--missing-policy-default`
  - malformed wrapper output uses `--malformed-bundle-default`
  - wrapper condition-evaluation errors in JSON output use
    `--condition-eval-error-default`
- For safer environments, set `--missing-policy-default request` and/or set
  `fallback_condition_eval_error` to `request` in emitted manifest generation.
- Use `--require-binary` to hard-fail hosts that cannot execute policy dispatch.

### Conditional export behavior

`sync_host_rules.py` can emit wrappers in two modes:

- default (legacy): only rule metadata is prepared; platform host rules are filtered to
  keep unconditional command controls unchanged.
- `--include-conditional`: also writes conditional controls through the command host
  surfaces for the selected platform(s).

Use this pattern to keep current host behavior while enabling wrapper-only evaluation:

```bash
python policy-contract/scripts/sync_host_rules.py \
  --policy-json /tmp/effective-policy.json \
  --out-dir /tmp/policy-host
```

To export conditional host fragments explicitly (during migration):

```bash
python policy-contract/scripts/sync_host_rules.py \
  --policy-json /tmp/effective-policy.json \
  --out-dir /tmp/policy-host \
  --include-conditional
```

### Go implementation

```bash
cd policy-contract/wrappers/go
go build -o policy-wrapper ./...
./policy-wrapper --bundle ./policy-wrapper-rules.json --command "git checkout main" --cwd /path/to/repo
```

### Rust implementation

```bash
cd policy-contract/wrappers/rust
cargo build --release
./target/release/policy-wrapper --bundle ./policy-wrapper-rules.json --command "git checkout main" --cwd /path/to/repo --json
```

### Zig implementation

```bash
cd policy-contract/wrappers/zig
zig build
./zig-out/bin/policy-wrapper --bundle ./policy-wrapper-rules.json --command "git checkout main" --cwd /path/to/repo --json
```

Notes:

- The dispatcher supports `allow|request|deny` exit codes:
  - `0` = allow
  - `1` = request
  - `2` = deny
- `policy-wrapper-dispatch.manifest.json` contains the exact script+bundle contract for
  host integration.
- The Zig binary reads the same schema and decision protocol as Go/Rust and emits
  `allow` by default when no conditional rule matches
